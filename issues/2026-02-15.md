# Points Language Issues — 2026-02-15

### Issue 11: No scoped/local verify mode

**Scenario**: After modifying a few spec points, running a full verify is slow and noisy — it checks the entire spec when only the changed points and their parent/child files need checking. The bidirectional code traceability (Format Rule 10) already provides the mapping needed to scope verification, but the verify action only supports full-spec or single-file mode.
**Proposal**: Add a `--local` flag to the verify action that takes changed spec points or files, uses Code Mapping to find affected source files, and checks only those points against their code + parent/child consistency.
**Resolution**: Added `--local` mode to the verify action with a 6-step scoped verification procedure.

### Issue 12: Workflow gate not enforced — spec and code changes bundled together

**Scenario**: The spec-first workflow rule (Action Rule 1) requires presenting spec changes for user approval before implementing code. In practice, the rule was written as a numbered list of steps, which the LLM reads once and then forgets during multi-step execution. Spec edits and code edits get bundled into a single pass without stopping for confirmation.
**Proposal**: Split the workflow into two explicit phases (Phase 1: spec-only edits, Phase 2: implementation after confirmation) with a HARD STOP between them that requires calling AskUserQuestion. Restrict Phase 1 to only editing `points/` files. Add rationalizations like "the spec change is trivial so I can bundle it with code" to the deny list.
**Resolution**: Updated workflow.md with Phase 1/Phase 2 structure, HARD STOP gate requiring AskUserQuestion, and expanded rationalization deny list.
