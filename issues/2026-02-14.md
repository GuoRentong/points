# Points Language Issues — 2026-02-14

### Issue 1: No code traceability mechanism

**Scenario**: Points captures "what" and "why" of a system design but provides no structured way to link points to actual source code (file paths, class names, function signatures). When code changes, there's no reverse lookup from code to affected points, and when points change, there's no forward lookup to affected code.
**Proposal**: Add a traceability annotation format, e.g. a `## Code Mapping` section or inline `@code(path/to/file.py:ClassName.method)` references. This is critical for the `evolve` and `sync` actions to work reliably at scale.

### Issue 2: No state or maturity tracking for points

**Scenario**: In human-agent coworking, different parts of the design are at different stages: some are sketches, some are reviewed, some are implemented and verified. Points has no way to express this. An agent reading the spec can't tell which parts are trustworthy and which are provisional.
**Proposal**: Consider a lightweight status annotation per component or file, e.g. `[draft]`, `[reviewed]`, `[implemented]`, `[verified]`. This enables prioritized verification and prevents agents from building on unreviewed foundations.

### Issue 3: Ambiguous multi-party dependency grouping criterion

**Scenario**: Rule 8 says "each inter-file dependency spec file should describe one minimal multi-party dependency" and "if they can be partitioned into multiple independent groups, do not mix them together." But "independent" is undefined. When multiple pairs share a common party (A-B, A-C, A-D), are they one multi-party group or three independent pairs? Both interpretations are defensible.
**Proposal**: Define "independent" precisely. One option: two dependency pairs are independent if removing one would not affect the description or correctness of the other. Under this definition, A-B and A-C can be grouped if they share a mechanism (e.g. ProxyBridge pattern) but should be split if they're coincidental.

### Issue 4: No change propagation tracking

**Scenario**: Rule 2 requires updating parent and child files on modification. For a large spec (50+ files), manually tracking which files need updating is error-prone. There's no mechanism to mark a point as "stale" or flag dependent files that need review after a change.
**Proposal**: Consider a lightweight dependency graph or "last-verified" timestamp mechanism. Alternatively, the `verify` action could be extended to detect staleness by comparing modification timestamps across parent-child pairs.

### Issue 5: No structured representation for invariants

**Scenario**: The "Definitions" layer mentions "invariants that must always hold" as a key concept, but invariants are written as regular points with no structural distinction. This means an agent running a trace has no way to automatically collect all invariants to check, and a verify pass can't distinguish "this is an invariant" from "this is a description."
**Proposal**: Add a structured invariant format, e.g. an `## Invariants` section per file or a naming convention for invariant points (e.g. `[INV]` prefix). This would make traces mechanically checkable.

### Issue 6: No trace storage format

**Scenario**: The "trace" action is defined but there's no format rule for how traces are stored, named, or organized. Are they files in the points/ directory? Separate directory? Inline in spec files? Without a defined format, traces are ephemeral — run once, lost, and never re-run. This undermines the philosophy that "traces are the mechanism that makes definitions trustworthy."
**Proposal**: Define trace storage: naming convention (e.g. `Trace-<scenario-name>.md`), required sections (scenario, steps, state changes, invariant checks, outcome), and a catalog mechanism so traces can be re-run after spec changes.

### Issue 7: No conflict resolution protocol for concurrent edits

**Scenario**: In human-agent coworking, multiple agents (or a human and an agent) may modify related points concurrently. Points has no merge strategy or conflict detection beyond git. Two agents could independently extend the same component's points with overlapping IDs or contradictory descriptions.
**Proposal**: Consider reserving ID ranges per agent session, or requiring a "lock" annotation when an agent is actively modifying a file. Alternatively, rely on git branching discipline but add a `verify --diff` mode that checks consistency of concurrent changes before merge.

### Issue 8: Intra-file dependencies section naming inconsistency

**Scenario**: Rule 7 and Rule 10 use the heading `## Intra-file Dependencies` for topic spec files. However, Rule 8 and the inter-file dependency spec example use just `## Dependencies` (without the "Intra-file" prefix). The SKILL.md example for `Database-1.Proxy.md` uses `## Intra-file Dependencies`, but the example for `Dep-1.1.ProxyManager-1.2.LoadBalancer.md` uses `## Dependencies`. This distinction is correct (inter-file dep files only have dependencies, not "intra-file" ones), but the rule text in Rule 10 only specifies the intra-file heading format. Rule 10 should explicitly state the heading format for inter-file dependency spec files as well.

### Issue 9: Rule 4 vs Rule 5 tension on unnamed points

**Scenario**: Rule 5 allows points to use "only the numeric suffix" when they "will not be further expanded into a topic-level or component-level spec." But Rule 4 requires parent-child redundancy where parent points correspond to child components. When a parent has unnamed points (per Rule 5) but the child file has named components, the redundancy chain is broken — the parent's `8.1.1` doesn't clearly map to the child's `8.1.1.YouTubeCrawlerState`.
**Proposal**: Clarify Rule 5 to state that points which correspond to child file components MUST use named IDs. The "numeric suffix only" option should be reserved for terminal points that will never become components in a child file.

### Issue 10: Language issues vs spec issues separation [RESOLVED]

**Scenario**: During verify, both spec consistency issues (missing files, ID errors, redundancy gaps) and language design observations were recorded in the same issues file. Action Rules 3-4 are scoped to language evolution, but the verify action's original step 7 ("Report any issues found") didn't distinguish between the two.
**Resolution**: Added Action Rule 5 to explicitly separate concerns. Updated verify steps 7-8 to clarify: spec issues are reported to the user; language observations go to the issues file. Updated config.md labels.
