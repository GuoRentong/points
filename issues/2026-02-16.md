# Points Language Issues — 2026-02-16

### Issue 13: Parent-child redundancy is mechanical busywork

**Scenario**: Format Rule 4 requires "intentional redundancy" between parent and child files for "chained recursive review." In practice, when a child component's implementation detail changes (e.g., "re-fetches from API" → "reads metadata.json from disk"), updating the parent summary is mechanical copy-paste that doesn't trigger genuine review. The parent summary tracks mechanism detail that changes frequently, creating sync burden without proportional value.

**Proposal**: Distinguish two kinds of points within any file:
1. **Points that expand into a child file** — the redundant overlap. In the parent, these should be stable purpose sentences describing *what* the component is, not *how* it works. The full evolving detail lives exclusively in the child file. Purpose sentences change only when a component is fundamentally restructured (rare).
2. **Points that don't expand** — leaf-level wherever they appear, even in "upper level" files. These carry full evolving detail as they do today.

This preserves the navigational overview value of parent files while eliminating ~90% of parent update burden. Example: `8.2.5.TopicDownload Per-topic content download.` (stable purpose) instead of `8.2.5.TopicDownload Reads metadata.json from disk (written during discovery) for topic type and content; falls back to API re-fetch if missing; parses two topic types...` (evolving mechanism detail).

**Resolution**: Added to Format Rule 4: "Parent points that expand into child files must not reference implementation details." Interpretation is left to the agent, with the human-agent review loop as the enforcement mechanism.

### Issue 14: "Format Rules" is a misleading section title

**Scenario**: Most rules in the "Format Rules" section are not about formatting — they define spec structure (file types, hierarchy, dependencies, traceability, proposals). Only rules 1, 3, and 12 are genuinely about format. Similarly, "Action Rules" duplicates content already in `workflow.md` (spec-first workflow, parent/child consistency) while mixing in language evolution tracking rules that belong with the workflow.

**Proposal**:
1. Rename "Format Rules" → "Spec Rules" in SKILL.md.
2. Move Action Rules into `workflow.md`: rules 1-2 are already covered by the workflow; rules 3-5 (language evolution tracking) merge into a new "When the Points language itself changes" section.
3. Replace the Action Rules section in SKILL.md with a one-line reference to `workflow.md`.
4. Update all internal references: "Format Rule X" → "Spec Rule X", "Action Rule X" → "see `workflow.md`".

**Resolution**: Renamed "Format Rules" to "Spec Rules". Moved Action Rules 3-5 into workflow.md as "When the Points language itself changes" section. Removed Action Rules section from SKILL.md, replaced with reference to workflow.md. Updated all cross-references in Actions section.
