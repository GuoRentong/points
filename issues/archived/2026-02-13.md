# Points Language Issues — 2026-02-13

## Issue 1: Format rules restructured [RESOLVED]

**Scenario**: During initial skill creation, format rules used 2-letter prefixes (e.g. `PX.1`, `OR.2.1`) and 2-space indentation.
**Proposal**: Switch to full hierarchical IDs (`Database-1.Proxy-1.ProxyManager`) with 3-space indentation.
**Resolution**: Adopted. Rules 3 and 4 split and rewritten.

## Issue 2: ID naming convention overhauled [RESOLVED]

**Scenario**: Original IDs were terse abbreviations (BC, OR, CI, PX, etc.) that were hard to read and didn't encode the hierarchy.
**Proposal**: Use `<ProjectName>-<TopicNumber>.<TopicName>-<ComponentNumber>.<ComponentName>` format with nested structure.
**Resolution**: Adopted. Rule 5 now fully specifies hierarchical ID scheme with shortened prefixes allowed within a file.

## Issue 3: Dependency structure formalized [RESOLVED]

**Scenario**: Original spec used a single `deps.md` file with flat bullet lists and a `## Relationships` section per file.
**Proposal**: Formalize into intra-file dependencies (within each spec file) and inter-file dependencies (separate `Dep-` prefixed files), both using two-level structure with globally unique dependency IDs.
**Resolution**: Adopted. Rules 7 and 8 added.

## Issue 4: Spec file naming convention added [RESOLVED]

**Scenario**: Original spec used short lowercase filenames (e.g. `proxy.md`, `backlog.md`).
**Proposal**: Use full topic ID as filename (e.g. `Database-1.Proxy.md`), `Root.md` for top-level, `Dep-` prefix for dependency files. Flat directory layout.
**Resolution**: Adopted. Rule 9 added.

## Issue 5: Spec text formatting rules added [RESOLVED]

**Scenario**: No explicit rules for markdown formatting of topics, components, and points within spec files.
**Proposal**: Level-1 heading for topic, `## Components` section, bold component IDs, 3-space indentation per level, `## Intra-file Dependencies` section.
**Resolution**: Adopted. Rule 10 added.

## Issue 6: Action rules added [RESOLVED]

**Scenario**: No rules governing the workflow of spec-first development, backward-tracing, or language evolution tracking.
**Proposal**: Add action rules requiring spec-first changes, parent/child recursive updates, issue logging, and changelog maintenance.
**Resolution**: Adopted as Action Rules 1–4.

## Issue 7: Example spec files section was empty [RESOLVED]

**Scenario**: The "Example Spec Files" section in SKILL.md was blank, making it hard to understand the format from rules alone.
**Proposal**: Add short examples for Root.md, a topic spec, and an inter-file dependency spec using the database example.
**Resolution**: Adopted. Three example files added.
